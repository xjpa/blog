## systematic program design

<mark>Step 1</mark>: figure out what we want our program to do -> requirement analysis with design recipe

- problem statement
- data definitions
- signature, purpose, stub
- examples/tests

<mark>Step 2</mark>: programs are too big to solve -> break down problem as per properties in design recipe

- for each sub-problem, apply the design recipe
- basically define the data structures, write a signature, purpose statement, and stub for each function handling a sub-problem
- understand how sub-problems are interdependent, think about the data flow

<mark>Step 3</mark>: write one part of the program for each problem

- useful programs need to be easily modified
- implement functions/modules for each subproblem, tackling one function (or a small set) at a time
- as each function is implemented, continuously test it against examples to make sure that theyre working in isolation
